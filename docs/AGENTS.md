# docs/ — 設計知識庫地圖

> `docs/` 只放**穩定、跨時間有效**的設計知識（為什麼 / 是什麼）。
> 「怎麼安裝、怎麼執行」屬於各 package 的 `README` 與 [guides/](guides/)。

## 子資料夾

| 路徑                           | 放什麼                                  | 何時讀 / 寫              |
| ------------------------------ | --------------------------------------- | ------------------------ |
| [BluePrint/](BluePrint/)       | 白皮書：協議願景、機制、數學模型        | 想理解「為什麼這樣設計」 |
| [specs/](specs/)               | 白皮書 → 程式碼的**橋接規格**（可實作） | 動工寫合約 / 服務前必讀  |
| [architecture/](architecture/) | 系統架構、資料流、時序圖、模組關係      | 想建立全局心智模型       |
| [adr/](adr/)                   | 架構決策紀錄（一決策一檔）              | 做重大技術選擇前後       |
| [api/](api/)                   | 後端 API 介面契約                       | 前後端協作               |
| [guides/](guides/)             | 開發環境、貢獻流程、部署                | 上手與日常操作           |

## 文件分工原則

| 文件                | 角色       | 回答               |
| ------------------- | ---------- | ------------------ |
| `docs/`             | 設計知識庫 | 為什麼？是什麼？   |
| 各 package `README` | 操作手冊   | 怎麼跑？怎麼測？   |
| `AGENTS.md`         | 導航地圖   | 東西在哪？去哪找？ |

> 寫文件前先問：這是「穩定設計知識」（放 docs）、「操作步驟」（放 README/guides）、還是「指標」（放 AGENTS.md）？
